ERP에서 학생 정보를 **실시간으로 연동**하여 등록/수정/비활성화를 자동화하고자 합니다.

## 3. 방식 A — Webhook (권장)

### 3-1. 트리거 이벤트

| 이벤트 | 발생 시점 | event 값 |
|---|---|---|
| 신규 학생 등록 | ERP에 수강생 생성 시 | `student.created` |
| 학생 정보 변경 | 이름/상태 변경 시 | `student.updated` |
| 학생 비활성화 | 수강 종료·환불·탈퇴 시 | `student.deactivated` |

### 3-2. Webhook 규격

- **Method**: `POST`
- **URL**: `https://interview.evenigame.com/api/webhooks/erp/student`
- **Content-Type**: `application/json`
- **보안 헤더**: `X-Signature` (공유 시크릿 키 기반 HMAC-SHA256 서명)

### 3-3. Payload 필드 (필수)

| 필드 | 타입 | 예시 | 비고 |
|---|---|---|---|
| `event` | string | `"student.created"` | 위 표의 event 값 중 하나 |
| `student_code` | string | `"EV20260042"` | **영문 대문자 + 숫자만, 3~20자** (하이픈/공백/소문자/한글 불가) |
| `name` | string | `"홍길동"` | 실명, 1~50자 |
| `is_active` | boolean | `true` | 수강 중(`true`) / 비활성(`false`) |
| `timestamp` | ISO8601 | `"2026-04-22T10:00:00+09:00"` | 이벤트 발생 시각 |

> **student_code 규격 매우 중요**
> - 정규식: `^[A-Z0-9]{3,20}$`
> - 면접앱에서 동일 정규식으로 검증하므로, ERP에서 발급 시점에 반드시 해당 규격을 준수해야 합니다.

### 3-4. Payload 예시

```json
{
  "event": "student.created",
  "student_code": "EV20260042",
  "name": "홍길동",
  "is_active": true,
  "timestamp": "2026-04-22T10:00:00+09:00"
}
```

### 3-5. 응답 및 재시도 정책

| 면접앱 응답 | 의미 | ERP 처리 |
|---|---|---|
| `200 OK` | 정상 처리 | 완료 |
| `400 Bad Request` | 페이로드 검증 실패 | **재시도 금지** (로그 기록 후 알림) |
| `401/403` | 서명 오류 | **재시도 금지** (비밀키 확인) |
| `5xx` / 타임아웃 | 서버 일시 장애 | **지수 백오프 3회** 재시도 후 DLQ 적재 |

- 타임아웃: 10초 권장
- 동일 이벤트 중복 전송 시에도 면접앱 측에서 멱등성 처리(upsert)

## 4. 방식 B — REST API (Pull, 대안)

Webhook 구축이 어려운 경우, 면접앱이 ERP를 주기적으로 조회하는 방식.

### 4-1. ERP 측 제공 엔드포인트

- **URL**: `GET /api/students?updated_after={ISO8601}&limit=500`
- **인증**: `Authorization: Bearer {API_KEY}` (ERP 팀에서 발급)
- **응답 포맷**:
```json
{
  "students": [
    {
      "student_code": "EV20260042",
      "name": "홍길동",
      "is_active": true,
      "updated_at": "2026-04-22T10:00:00+09:00"
    }
  ],
  "next_cursor": null
}
```
- **Pagination**: `next_cursor` 또는 `offset` 방식
- **Rate limit**: 면접앱은 5~10분 주기로 호출 (분당 1회 이하)
